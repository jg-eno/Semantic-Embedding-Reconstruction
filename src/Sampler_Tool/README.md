# ReLoDer Sampler

A tool for sampling multiple text reconstructions from a single paragraph embedding, built on top of the ReLoDer (Reconstruction via LoRA Decoder) inversion model.

---

## Why this exists

The obvious question: if you want *n* semantically similar sentences, why not call an LLM API with a high temperature and a prompt like *"rephrase this 5 times"*?

Because that is a fundamentally different operation.

An LLM samples from `P(token | prompt)` — it is exploring its own language prior, conditioned loosely on your instruction. Similarity between outputs is incidental. There is no guarantee the rephrased sentences are semantically equivalent in any measurable sense, and there is nothing anchoring them to a specific point in representation space.

This tool does something different: it inverts a **fixed embedding vector**.

The encoder maps your sentence to a single point `e` in embedding space. The inverter's job is to find text `t` such that `encode(t) ≈ e`. Running the decoder `n` times with stochastic sampling produces `n` distinct surface forms that all correspond to the same embedding. The semantic content is not controlled by a temperature knob or a prompt — it is locked by the geometry of the embedding space itself.

This means:

**The outputs are preimages of a single embedding, not rephrases of a sentence.**

That distinction matters in several concrete ways:

### 1. Controlled semantic augmentation

When training models that consume embeddings (retrievers, classifiers, rerankers), augmenting with LLM rephrases introduces uncontrolled semantic drift — the paraphrases may shift the meaning in ways you cannot measure without re-embedding. This tool guarantees all augmented samples map back to the same neighbourhood in embedding space, because they were generated by inverting exactly that point. The augmentation is anchored.

### 2. Embedding space coverage analysis

Given an embedding `e`, the `n` outputs form an empirical sample of the **preimage set** `{t : encode(t) ≈ e}`. This reveals what the encoder treats as equivalent. Large preimage sets indicate the encoder is lossy at that point. Small, tightly clustered preimage sets indicate high-fidelity encoding. This is a diagnostic tool for understanding what information a specific embedding model preserves and what it discards.

### 3. Adversarial and robustness testing

Any system that relies on embeddings for identity — deduplication, plagiarism detection, semantic search — can be probed by generating multiple preimages of a target embedding. If two semantically distinct texts hash to the same embedding, this tool will surface both. That is not a paraphrase — it is a collision.

### 4. Reconstruction fidelity benchmarking

The canonical use case for inversion research: given an embedding, can you recover the original text? Running `n` samples and measuring how often the original sentence appears (or how close the nearest sample is) quantifies how much information the embedding retains. This is the metric the ReLoDer model is trained to maximise, and this tool is the evaluation harness.

## How it works

```
sentence
   │
   ▼
Encoder (Qwen3-Embedding-0.6B)
   │  mean-pool + L2-normalise
   ▼
embedding e ∈ ℝ¹⁰²⁴   ◄── fixed point in embedding space
   │
   ▼
prefix_len × SingleTokenMLP     each MLP projects e → one soft token
   │  stack → (1, prefix_len, decoder_hidden)
   ▼
LoRA-fine-tuned Qwen3-0.6B decoder
   │  autoregressive decode with temperature T_i = T₀ + i·Δ
   ▼
n candidate reconstructions
   │
   ▼
Jaccard deduplication + greedy max-dissimilarity ranking
   │
   ▼
top-n diverse outputs
```

The temperature schedule `T_i = T₀ + i·Δ` means the first sample is the decoder's most confident reconstruction, and each subsequent sample explores a progressively wider region of the output distribution — all still conditioned on the same embedding.

```bash
# clone and install
```

## Configuration

All defaults live in `config.py`.

| Constant | Default | Description |
|---|---|---|
| `CONFIG["repo"]` | `Subhav-K/ReLoDer_v4` | HuggingFace checkpoint repo |
| `CONFIG["filename"]` | `best_steps_checkpoint.pt` | Checkpoint file inside the repo |
| `CONFIG["prefix_len"]` | `64` | Soft prefix length (auto-detected from checkpoint) |
| `ENCODER_MODEL_NAME` | `Qwen/Qwen3-Embedding-0.6B` | Sentence encoder |
| `DECODER_BASE_NAME` | `Qwen/Qwen3-0.6B` | Base LM for the inverter decoder |
| `MAX_NEW_TOKENS` | `20` | Max tokens generated per sample |
| `SAMPLER_DEFAULTS["temperature"]` | `1.4` | Base sampling temperature T₀ |
| `SAMPLER_DEFAULTS["temperature_step"]` | `0.05` | Per-sample temperature increment Δ |
| `SAMPLER_DEFAULTS["top_p"]` | `0.95` | Nucleus sampling |
| `SAMPLER_DEFAULTS["top_k"]` | `50` | Top-k candidate pool |
| `SAMPLER_DEFAULTS["repetition_penalty"]` | `1.3` | Penalise repeated tokens |

## Programmatic usage

```python
from sampler import Sampler

sampler = Sampler()
result = sampler.sample("The speed of light is constant in a vacuum.", n=5)

for text in result.samples:
    print(text)

# result.embedding holds the (1, 1024) CPU tensor if you need it
```

```python
# batch
results = sampler.sample_batch([
    "The Earth orbits the Sun.",
    "Water boils at 100 degrees Celsius.",
], n=5)
```

## Memory requirements

| GPU VRAM | Status |
|---|---|
| 4 GB (e.g. RTX 3050 Laptop) | Works — sequential encode/invert with encoder teardown between phases |
| 8 GB | Both models resident simultaneously; remove encoder teardown in `sampler.py` |
| 16 GB+ | Full batch inference; increase `n`, `MAX_NEW_TOKENS`, `prefix_len` |

On a 4 GB GPU the tool uses a two-phase memory strategy: the encoder runs first and is fully destroyed (not just offloaded) before the inverter loads. See `CLAUDE.md` for the full breakdown.

## Checkpoint

The current weights are `Subhav-K/ReLoDer_v4`, trained to invert embeddings from `Qwen/Qwen3-Embedding-0.6B`. Architecture parameters (`prefix_len`, `mlp_hidden_dim`, LoRA rank, target modules) are all auto-detected from the checkpoint at load time — no manual configuration required when switching checkpoints.

## Scaling to a larger GPU

When moving to a GPU with more VRAM (e.g. RTX 3090 24 GB, A100 40/80 GB):

### Parameters to increase

| Parameter | Where | Current (4 GB) | Suggested (24 GB+) |
|---|---|---|---|
| `prefix_len` | `config.py` → `CONFIG["prefix_len"]` (or retrain with more) | 64 | 128–256 |
| Batch size in `Encoder.encode` | pass a list to `sampler.sample_batch` | 1 | 8–32 |
| `MAX_NEW_TOKENS` | `config.py` | 20 | 128–512 |
| `n` samples per call | CLI / UI | 5 | 20–50 |
| MLP `mlp_hidden_dim` | checkpoint-dependent, auto-detected | 4096 | 4096–8192 |

### Things to remove or relax

- The two-phase encode→destroy→invert sequence is unnecessary — both models can stay resident simultaneously
- `torch_dtype=fp16` on the decoder can be dropped; train/infer in bf16 or fp32 for better stability
- The `dedup_threshold=0.85` can be tightened (e.g. 0.7) when generating more candidates at higher n
- `temperature_step` can be reduced (e.g. 0.02) when n is large, to keep the schedule from going too hot

### Code change to keep both models resident (large GPU)

In `sampler.py`, remove `_destroy_encoder()` and simplify `_load_inverter`:

```python
def _load_inverter(self, paragraph_dim: int) -> None:
    if self._inverter is None:
        print("[Sampler] Loading inverter …")
        self._inverter = InverterFactory().load(
            repo=self.cfg["repo"],
            filename=self.cfg["filename"],
            paragraph_dim=paragraph_dim,
        )
        self._tokenizer = self._encoder.tokenizer
```
